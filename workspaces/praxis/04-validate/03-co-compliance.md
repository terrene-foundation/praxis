## Detailed Findings

### Finding 1: Layer 5 (Learning) is unimplemented

**Severity**: HIGH (architectural completeness)

The CO five-layer architecture defines Learning as the fifth layer. Praxis defines the schema for it (domain YAMLs include `capture.observation_targets` and `capture.auto_capture`), but no runtime code reads these fields or implements:

- Observation logging (JSONL or equivalent)
- Pattern detection across sessions
- Instinct formation with confidence scoring (frequency 40% + success 30% + recency 20% + consistency 10%)
- Evolution suggestions (Skill at >=0.7, Command at >=0.6, Agent at >=0.8)
- Human-gated formalization

**Impact**: Without Layer 5, Praxis does not compound knowledge across sessions. Each session starts from the same baseline. This is acceptable for v0.1.0 but must be addressed before claiming full CO conformance.

**Recommendation**: Implement an `ObservationCapture` module that logs deliberation patterns, constraint events, and workflow transitions to JSONL. Build pattern analysis on top. The domain YAML `observation_targets` already define what to observe -- the pipeline to capture and analyze is what is missing.

### Finding 2: Phase transitions are metadata, not enforced

**Severity**: MEDIUM (Layer 4 completeness)

Domain YAMLs define phases with `approval_gate: true/false`, but the runtime has no phase state machine. There is no code that:

- Tracks which phase a session is currently in
- Prevents advancing past an approval gate without human approval
- Prevents skipping phases
- Records phase transitions in the deliberation log

**Impact**: The Instructions layer (Layer 4) defines what should happen but does not enforce it. A client could skip the analysis phase and go straight to implementation without any runtime objection.

**Recommendation**: Add a `phase` field to session state. Add a `PhaseManager` that enforces the transition graph defined in domain YAMLs. Approval gates should create HELD actions that must be resolved before the session can advance.

### Finding 3: Bainbridge's Irony has no proactive defense

**Severity**: MEDIUM (Principle 6 completeness)

The implementation provides the building blocks (constraint reasons, deliberation rationale, approval gates) but has no proactive mechanism to detect or counter the specific risk Bainbridge identifies: human operators losing capability because they intervene too rarely.

No code exists for:

- Tracking the ratio of AUTO_APPROVED to HELD actions over time
- Detecting approval fatigue (rapid approval of HELD actions without deliberation)
- Periodic spot-check elevation (randomly HOLDING an otherwise AUTO_APPROVED action to keep the human engaged)
- Alerting when a human's approval latency drops below a meaningful review threshold

**Recommendation**: Add an `ApprovalHealthMonitor` that tracks approval patterns per session and per human. Flag sessions where HELD actions are being approved faster than reasonable review time. Consider a configurable spot-check rate (e.g., 5% of AUTO_APPROVED actions elevated to HELD).

### Finding 4: In-memory storage with persistence models defined but not wired

**Severity**: LOW (infrastructure, not methodology)

`SessionManager._sessions` is a Python dict. `persistence/models.py` defines DataFlow models for Session, DeliberationRecord, ConstraintEvent, TrustChainEntry, and Workspace with proper indexes. `persistence/queries.py` provides the query interface. But the actual write path from core modules to DataFlow is not connected.

**Impact**: Sessions survive within a process but not across restarts. Export/import provides a manual workaround. This is a plumbing issue, not a methodology issue -- the architecture is correct, the wiring is incomplete.

**Recommendation**: Wire `SessionManager`, `DeliberationEngine`, and `ConstraintEnforcer` to write through to DataFlow models. This is straightforward given that the models and query layer already exist.

### Finding 5: Constraint enforcement depends on caller discipline

**Severity**: LOW (architectural note)

`ConstraintEnforcer.evaluate()` returns a verdict but does not block execution. It is the caller's responsibility to check the verdict and refuse to proceed when BLOCKED. This is architecturally sound (separation of evaluation from execution), but means every call site must implement the check correctly.

**Impact**: If a new API endpoint or CLI command forgets to check the verdict, a BLOCKED action could proceed. This is a standard software correctness concern, not a methodology gap.

**Recommendation**: Consider a `enforce()` method that raises on BLOCKED verdicts, as a convenience for callers who want automatic blocking. The current `evaluate()` method should remain for callers who need to inspect the verdict before acting.

## Overall CO Conformance

**Verdict: GENUINE CO IMPLEMENTATION -- Layers 1-4 are strong, Layer 5 is absent, two principles are partial.**

Praxis is a genuine implementation of CO methodology, not governance theater. The evidence:

1. **Trust infrastructure is real cryptography, not logging.** Ed25519 signatures, JCS-canonical hashing, hash-chained audit anchors, verification bundles with client-side SubtleCrypto verification. This is not "we record what happened" -- it is "we can cryptographically prove what happened and any tampering is detectable."

2. **Constraint enforcement is deterministic, not advisory.** The gradient engine is a pure function with normative thresholds. BLOCKED means blocked. Constraint tightening is mathematically enforced (subset checks, less-than-or-equal checks). Delegation cannot expand parent authority.

3. **Five dimensions are consistently enforced.** Every evaluation checks all five dimensions (Financial, Operational, Temporal, Data Access, Communication) and takes the worst-case verdict. No shortcutting, no partial evaluation.

4. **Domain differentiation is substantive.** Six domains with meaningfully different constraint templates, workflow phases, and capture rules. COE's progressive autonomy (year1->year2->year3 constraints loosen as students develop capability) is a particularly strong demonstration that domains encode genuine institutional knowledge, not boilerplate.

5. **The verification bundle is production-grade.** A self-contained ZIP that opens in any browser and verifies the trust chain client-side with zero installation is exactly what independent third-party verification requires.

**What prevents a full PASS:**

- Layer 5 (Learning) is entirely absent from the runtime. The schema exists but no code implements observation capture, pattern detection, or knowledge compounding.
- Phase transitions are not enforced at runtime, weakening Layer 4.
- No proactive Bainbridge defense (approval fatigue detection, spot-check elevation).

**For v0.1.0, this is strong.** The trust infrastructure is the hardest part to get right, and Praxis gets it right. The missing pieces (Layer 5, phase enforcement, Bainbridge monitoring) are additive -- they build on the existing architecture without requiring restructuring.
